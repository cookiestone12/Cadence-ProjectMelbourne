Replit Agent Prompt – “Gotcha Catalog Manager” Fusion App

You are a senior full-stack TypeScript engineer.
You are working in a repo that already contains the Co5 Manager Admin Platform (React + Express + Postgres + Drizzle) and you have a conceptual white paper for a Gotcha rights-discovery platform.

Your task: turn this into a single demo app called “Gotcha Catalog Manager” that:
	•	Feels visually and structurally similar to Apple Music / Apple Music for Artists.
	•	Supports multi-tenant catalog management for labels, publishers, production companies, and individual creators.
	•	Combines:
	•	Co5’s checklist + health score + placement tracking
	•	Gotcha’s Profiles/Works / creator-centric catalog view
	•	Sets up clean hooks for a future valuation module (Ampersound / Luminate-based) but does not implement valuations yet.

This is a demo: payment, Luminate integration, and external APIs are not implemented, but the architecture must be ready for them.

⸻

1. High-Level Product Requirements

Implement the app so that:
	1.	A label/publisher/production company user (org owner) can:
	•	Create an Organization.
	•	Add creators to their roster (artists, songwriters, producers).
	•	Upload and manage catalog data (songs, features, productions) across the roster.
	•	Track placement status and health for each song (contracts sent, executed, registered, invoiced, paid, etc.).
	•	Click into any creator → see their catalog → click a song → see all song details.
	•	Export a Schedule A for a creator (CSV for demo).
	2.	An individual creator (artist/songwriter/producer) can:
	•	Create a personal workspace (Organization with one member is fine).
	•	See their own catalog in an Apple Music–style UI.
	•	Click into songs and see admin status and external DSP links.
	3.	Song detail pages show:
	•	Credits (writers, artists, producers, publishers).
	•	Placement status + checklist/health score (Co5 features).
	•	Links to Apple Music / Spotify (stored as URLs).
	•	Basic rights metadata (ISRC, ISWC, PRO/IPI fields ready for future Gotcha integration).
	4.	The app is structured so you can later bolt on:
	•	A valuation module (separate service / section in the UI) that uses Luminate.
	•	Additional rights-discovery functionality pulling from Gotcha’s model.

⸻

2. Domain Model & Database (Drizzle + Postgres)

Extend/adjust the existing Co5 schema into the following entities. Keep existing tables where possible; add new ones as needed. Use Drizzle migrations, don’t wipe existing demo data.

Core tenancy & users
	•	organizations
	•	id
	•	name
	•	type ('LABEL' | 'PUBLISHER' | 'PRODUCTION_COMPANY' | 'MANAGER' | 'INDIVIDUAL')
	•	created_at, updated_at
	•	users
	•	Keep existing user table.
	•	Add relationship via a join table:
	•	organization_members
	•	id
	•	organization_id (FK → organizations)
	•	user_id (FK → users)
	•	role ('OWNER' | 'ADMIN' | 'MEMBER')
	•	created_at

Creators (roster)
	•	creators
	•	id
	•	organization_id (FK)
	•	display_name
	•	legal_name (nullable)
	•	roles (array/string enum: 'ARTIST' | 'SONGWRITER' | 'PRODUCER')
	•	primary_territory (nullable)
	•	primary_pro (nullable; 'ASCAP' | 'BMI' | 'PRS' | 'SESAC' | 'OTHER')
	•	primary_ipi (nullable)
	•	hero_image_url (nullable; used for Apple Music style header)
	•	created_at, updated_at

This is essentially Gotcha profiles scoped to an organization.

Songs / Works

Update/extend the Co5 songs table into a more general catalog song:
	•	songs
	•	id

	•	title
	•	primary_artist (string)
	•	isrc (nullable)
	•	iswc (nullable)
	•	project_title (nullable)
	•	release_date (nullable date)
	•	status_health_score (0–100, derived from checklist)
	•	has_contract_sent (bool)

	•	is_registered_with_pro (bool)
	•	is_registered_with_dsp (bool)
	•	is_invoiced (bool)

	•	created_at, updated_at
	•	song_credits
	•	id
	•	song_id (FK)
	•	creator_id (FK → creators)
	•	role ('ARTIST' | 'FEATURED_ARTIST' | 'SONGWRITER' | 'PRODUCER' | 'MIX_ENGINEER' | 'OTHER')
	•	share_percentage (nullable)
	•	song_dsp_links
	•	id
	•	song_id (FK)
	•	platform ('APPLE_MUSIC' | 'SPOTIFY' | 'YOUTUBE_MUSIC' | 'OTHER')
	•	url
	•	created_at

Placements / checklist / health (Co5 logic)

Re-use or adapt existing Co5 tables:
	•	checklist_items
	•	Existing codes like AD-01, LG-01, MD-01, DSP-01, SY-01, etc.
	•	Category ('ADMIN' | 'LEGAL' | 'METADATA' | 'DSP' | 'SYNC' | 'PAYMENT')
	•	Weight (int)
	•	Description
	•	song_checklist_status
	•	id
	•	song_id
	•	checklist_item_id
	•	status ('NOT_STARTED' | 'IN_PROGRESS' | 'COMPLETED' | 'BLOCKED')

Schedule A exports

No separate table is required, but ensure the schema above supports:
	•	For a given creator, you can fetch:
	•	All songs where they appear in song_credits.
	•	Shares and roles, plus ISRC/ISWC and metadata.
	•	This can be exported as a CSV from the API.

Valuation hooks (future)

Add a small table to prepare for valuations (no full engine now):
	•	song_valuation_snapshots
	•	id
	•	song_id

	•	source ('MANUAL' | 'LUMINATE' | 'EXTERNAL')
	•	snapshot_date
	•	notes (text, nullable)

You don’t need UI for this yet; just the table and simple read/write endpoints.

3. Backend API (Express)

Refactor/extend routes into something like:
	•	Auth remains as is (JWT).
	•	Every request is scoped to an organization_id derived from current user context.

Organization & membership
	•	GET /api/orgs/current → returns the active org for the logged-in user (for demo, pick first membership).
	•	GET /api/orgs/:id/creators
	•	POST /api/orgs/:id/creators
	•	GET /api/creators/:id → detail, with aggregated stats (song count, placements, average health).

Catalog & songs
	•	GET /api/orgs/:id/songs with filters:
	•	creatorId, role, status, healthRange, pagination.
	•	GET /api/songs/:id
	•	Include: song metadata, credits, checklist status, health, DSP links.
	•	POST /api/orgs/:id/songs/import
	•	Re-use Co5 Excel/CSV upload/import pipeline.
	•	PATCH /api/songs/:id
	•	Update song fields and checklist statuses.

Credits & DSP links
	•	POST /api/songs/:id/credits
	•	DELETE /api/songs/:id/credits/:creditId
	•	POST /api/songs/:id/dsp-links
	•	DELETE /api/songs/:id/dsp-links/:linkId

Checklist & health
	•	Re-use Co5 logic to compute health:
	•	A service that takes all song_checklist_status records and item weights and computes 0–100.
	•	Expose:
	•	GET /api/songs/:id/checklist
	•	PATCH /api/songs/:id/checklist (update statuses in batch).

Schedule A export
	•	GET /api/creators/:id/schedule-a
	•	Returns CSV (for now) with:
	•	Song Title
	•	Role
	•	Share % (if stored)
	•	ISRC
	•	ISWC
	•	DSP links (Apple/Spotify URLs)
	•	Status flags (registered/paid etc.)
	•	Implement as a simple CSV response; no fancy PDF needed yet.

Valuation stubs
	•	GET /api/songs/:id/valuation
	•	For now, just returns latest song_valuation_snapshots row if any, or null.
	•	POST /api/songs/:id/valuation
	•	Admin-only, allow manual entry of a valuation.
	•	Add a comment/TODO in the valuation service: this will later call Luminate/Ampersound.
