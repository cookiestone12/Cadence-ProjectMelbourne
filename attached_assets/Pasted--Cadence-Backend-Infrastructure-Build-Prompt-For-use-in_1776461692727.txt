# Cadence — Backend Infrastructure Build Prompt

# For use in Replit AI / Agent

-----

## WHO WE ARE

Cadence (Cadence Catalog Intelligence Co.) is a multi-tenant SaaS platform for music labels,
publishers, production companies, and independent creators. The platform handles music catalog
management, royalty processing, rights intelligence, sync placement pipelines, and catalog
valuation. This is a real, revenue-generating product approaching its beta launch with paying
organizations.

-----

## WHAT WE ARE BUILDING

We are building the production backend infrastructure for Cadence. This is not a prototype, not
an intern tool, and not a temporary layer. This is the canonical backend that the company will
operate on going forward — serving internal staff, interns, clients, and eventually a full
engineering team. It must be architected with the discipline and care of a production system
from day one.

The existing codebase is a FastAPI + PostgreSQL application running inside Replit. We are NOT
migrating off Replit at this time. What we are doing is:

1. Exposing the backend through a stable, persistent, publicly accessible HTTPS URL using
   Replit’s native deployment infrastructure
1. Ensuring that URL is the single point of access for all backend operations — no one needs
   Replit workspace access to use, develop against, or operate this backend
1. Implementing proper access control, environment separation, API documentation, and
   observability so the system is operable by a team
1. Architecting every layer in a way that makes a future migration to AWS, GCP, or equivalent
   infrastructure straightforward — clean environment abstraction, no platform lock-in in the
   application code

-----

## CURRENT ARCHITECTURE

### Runtime

- Framework: FastAPI (Python)
- Server: Gunicorn with Uvicorn workers
- Entry point: run_backend.sh — runs db_setup.py (Alembic migrations + seed), then starts
  Gunicorn
- Frontend: React 18 + Vite, built to static assets, served by FastAPI StaticFiles mount

### Database

- Engine: PostgreSQL
- Tables: 80 (grown from original 66)
- ORM: SQLAlchemy, all models in backend/models/models.py
- Migrations: Alembic, 5 revisions on disk, latest revision ID: a7b9c2d4e5f6
- Single database instance — no staging DB currently configured

### Routes

- 49 route modules under backend/routes/, one file per domain
- Representative modules: auth.py, songs.py, catalog.py, creators.py, works.py, releases.py,
  royalties.py, royalty_processing.py, contracts_mgmt.py, placements.py, brief_builder.py,
  audio.py, integrations.py, spotify_import.py, client_portal.py, client_sharing.py,
  tenant_admin.py, admin.py, analytics.py, valuations.py, valuation_reports.py,
  registration_reports.py, schedule_a.py, exports.py, actions.py, notifications.py, push.py,
  audit_log.py, bulk.py, settings.py, organizations.py, storage_scan.py, csv_upload.py,
  account_links.py, creative_directory.py

### Services

- backend/services/ contains all business logic and external integrations
- Key services: royalty_processing_engine.py, valuation_engine.py, decay_analytics_engine.py,
  contract_parser.py, document_parser.py, classification_engine.py, scoring_engine.py,
  reconciliation_engine.py, spotify_service.py, storage_service.py, scan_service.py,
  email_provider.py, audit_service.py, chartmetric_service.py, luminate_service.py

### Authentication

- JWT via PyJWT, bcrypt password hashing
- Token expiration: 24 hours (ACCESS_TOKEN_EXPIRE_MINUTES = 60 * 24)
- No refresh tokens — re-login on expiration
- Auth dependency: get_current_user in backend/utils/auth.py

### Authorization — Two-Layer Role Model

- Platform level: User.is_master_admin boolean — full cross-org access
- Organization level: OrganizationMemberRole enum on OrganizationMember
  - OWNER: full org control, billing, member management
  - ADMIN: full feature access within org
  - MEMBER: standard access within org
- Additional: ClientShareRole (cross-org client portal access), CreatorContactRole
  (contact-to-creator linking)

### Environment Variables

- Stored in Replit Secrets, accessed via os.getenv — no .env files, nothing hardcoded
- Active secrets: DATABASE_URL, AI_INTEGRATIONS_OPENAI_API_KEY, DROPBOX_APP_KEY,
  DROPBOX_APP_SECRET, LASTFM_API_KEY, YOUTUBE_API_KEY, VAPID_PRIVATE_KEY, PGDATABASE,
  and standard Postgres helper vars
- Resend (email) and Spotify credentials flow through the platform’s integration system

### Logging

- Structured JSON to stdout via backend/utils/logging_config.py
- Logger name: “cadence”
- Custom JSONFormatter — Replit’s deployment log viewer captures all output
- Uvicorn access logs suppressed to WARNING
- In-app audit_log system (AuditLog table) for user-facing action tracking
- No external error tracker (Sentry, Datadog) currently wired

### Background Jobs

- APScheduler configured in backend/services/email_scheduler.py
- Email digest job: every 15 minutes
- Storage scan job: every hour

### PWA

- Service worker, web manifest, push notifications via pywebpush (VAPID)

### COMPONENT 1: REPLIT NATIVE DEPLOYMENT WITH STABLE PUBLIC URL

Configure Replit’s native Deployments so the application runs on a persistent HTTPS URL
that stays live regardless of whether the Replit workspace is open. This URL becomes the
canonical access point for all backend consumers — internal staff, API clients, frontend,
and future engineering team.

Requirements:

- Use Replit Autoscale or Reserved VM deployment (choose the appropriate type for a
  persistent FastAPI + PostgreSQL application — do not use Static deployment)
- The deployed instance must serve both the FastAPI API and the Vite-built frontend static
  assets, exactly as the current run_backend.sh flow does
- Run Alembic migrations on every deploy before the server starts — db_setup.py already
  handles this; ensure the deploy hook calls it
- The deployment URL must be stable and not change between deploys
- Add a .replit or replit.toml deployment configuration block that makes the deploy
  reproducible and explicit — document what each field does in an inline comment

Custom Domain:

- Read ALLOWED_ORIGINS from Replit Secrets as a comma-separated string
- Parse it in backend/main.py and pass to CORSMiddleware — never hardcode domain strings
- Add both the Replit deployment URL and any custom domain (e.g., api.cadence-ci.com) to
  ALLOWED_ORIGINS in Replit Secrets
- Document in a DEPLOYMENT.md file exactly how to point a Google Domains DNS record to the
  Replit deployment URL (A record vs CNAME, which value to use, TTL recommendation)

Health Check:

- Add GET /health — unauthenticated, returns:
  {
  “status”: “ok”,
  “version”: “<git commit hash or build tag from env>”,
  “db”: “connected” | “unreachable”,
  “timestamp”: “<ISO 8601 UTC>”
  }
- The db field must actually attempt a lightweight DB query (SELECT 1) and report the
  real connection status — do not hardcode “connected”
- This endpoint will be used for uptime monitoring and future load balancer health checks

### COMPONENT 2: ENVIRONMENT SEPARATION (DEVELOPMENT vs PRODUCTION)

The current setup uses a single database for both development and production. Implement
a proper environment model so development work never touches live client data.

Requirements:

- Add an APP_ENV secret to Replit Secrets with values: “development” | “production”
- When APP_ENV=production:
  - Enforce stricter CORS (only whitelisted origins)
  - Disable any debug routes, test endpoints, or data seed utilities
  - Suppress detailed error stack traces in API responses (return generic error messages,
    log the full trace internally)
  - Require HTTPS (reject HTTP requests)
- When APP_ENV=development:
  - Allow permissive CORS for local development
  - Include detailed error responses
  - Allow data seed and reset utilities

Staging Database:

- Document in DEPLOYMENT.md how to provision a second PostgreSQL instance (via Replit
  Database, Supabase free tier, or Neon) to serve as the staging database
- Add STAGING_DATABASE_URL to Replit Secrets
- When APP_ENV=development, the application should connect to STAGING_DATABASE_URL if
  present, falling back to DATABASE_URL
- The Alembic env.py should read the correct DATABASE_URL based on APP_ENV
- Include a one-time script: scripts/seed_staging.py that:
  - Connects to STAGING_DATABASE_URL
  - Runs all Alembic migrations
  - Seeds a sandbox organization with test users at every role level (OWNER, ADMIN, MEMBER)
  - Seeds 20 synthetic songs, 5 creators, 2 royalty statements, and 1 contract
  - All seed data is clearly namespaced (org name: “Cadence Sandbox”, usernames prefixed
    with “sandbox_”) so it can never be confused with production data

### COMPONENT 3: TEAM ACCESS CONTROL

The current role system (OWNER, ADMIN, MEMBER, is_master_admin) serves client-facing use
cases. Build an additional internal team access layer for Cadence staff.

Requirements:

Internal Staff Role:

- Add a new boolean field to the User model: is_cadence_staff — default False
- is_cadence_staff grants access to all organizations’ data in read mode
- is_cadence_staff does NOT grant write access to client data unless is_master_admin is
  also True
- Add a new Alembic migration for this field — do not modify the existing migration files
- Update get_current_user and the relevant authorization middleware to recognize this role

Staff User Provisioning:

- Add a protected route POST /api/internal/provision-staff-user (requires is_master_admin)
  that creates a user account with is_cadence_staff=True
- This route accepts: { “username”: str, “email”: str, “password”: str, “role_note”: str }
- role_note is a freeform string logged to AuditLog explaining who this user is and why
  they were provisioned
- On creation, send a welcome email via the existing email_provider.py with their login
  credentials and the backend URL

Staff User Deprovisioning:

- Add a protected route POST /api/internal/deprovision-staff-user (requires is_master_admin)
  that accepts { “user_id”: int } and:
  - Sets is_cadence_staff=False
  - Invalidates all active sessions (requires a session tracking mechanism — see below)
  - Logs the deprovisioning to AuditLog with timestamp and actor

Session Tracking (required for deprovisioning):

- Add a UserSession table:
  - id, user_id (FK), token_hash (SHA-256 of JWT), created_at, expires_at, is_revoked,
    ip_address, user_agent
- On every login (POST /api/auth/login), create a UserSession record
- In get_current_user, after validating the JWT signature and expiration, check that the
  token_hash exists in UserSession and is_revoked=False — if not, return 401
- On deprovisioning, set is_revoked=True on all UserSession records for that user_id
- Add a nightly cleanup job (APScheduler) that deletes UserSession records older than
  30 days with is_revoked=True or past expires_at

### COMPONENT 5: INTERNAL ADMIN PORTAL

Build a web-based internal portal served at /internal — a clean, professional interface
through which Cadence staff can manage the platform without touching Replit or raw SQL.
This portal is distinct from the existing /admin (TenantAdminPage) and /super-admin
(AdminDashboard) pages, which are client-facing.

The internal portal is for Cadence company operations: onboarding new client organizations,
monitoring platform health, managing staff accounts, and accessing database-level views.

Portal Authentication:

- /internal requires is_cadence_staff=True or is_master_admin=True
- Users who are not authorized are redirected to /internal/login
- The login form at /internal/login uses the existing JWT auth system (POST /api/auth/login)
- On successful login, JWT is stored in httpOnly cookie (not localStorage) for this portal

Portal Pages and Capabilities:

/internal/dashboard

- Platform health overview:
  - Total organizations (active / inactive)
  - Total users (all orgs combined)
  - Total songs in the catalog (all orgs)
  - Total royalty statements processed (last 30 days)
  - API health status (live call to GET /health)
  - Database connection status
  - Background job status (last run timestamps for email digest and storage scan jobs)
- Recent audit log entries (last 50 actions across all orgs, with org name, user, action,
  timestamp)

/internal/organizations

- Full list of all organizations on the platform (name, type, account_type, member count,
  song count, created_at)
- Search and filter by org name, type, status
- Click into any org to view:
  - All members with roles
  - Song count, creator count, contract count
  - Recent audit log entries for that org
  - Option to impersonate the org (view the platform as that org’s OWNER would — read-only)
- Ability to create a new organization directly from this interface (calls existing
  POST /api/organizations endpoint)
- Ability to assign an access_code to an organization

/internal/users

- Full list of all users across all organizations (username, email, org memberships, roles,
  is_cadence_staff, is_master_admin, last_login)
- Search by username or email
- Provision new staff user (calls POST /api/internal/provision-staff-user)
- Deprovision staff user (calls POST /api/internal/deprovision-staff-user)
- View active sessions per user

/internal/database

- A read-only database browser:
  - Lists all 80 tables
  - Click any table to see its columns, types, and a paginated view of rows (default: 50 rows,
    max: 200 rows per page)
  - Filter rows by any column value
  - Export table view as CSV
- This is strictly read-only — no direct database writes from this interface
- All queries are executed via SQLAlchemy through the existing get_db dependency — no raw
  psycopg2 connections, no SQL string injection risk
- Every table access is logged to AuditLog with action=“INTERNAL_DB_VIEW”

/internal/logs

- Streaming view of the cadence structured JSON logs from stdout
- Filter by log level (INFO, WARNING, ERROR)
- Filter by timestamp range
- Display as a clean, readable table (timestamp, level, logger, message, any extra fields)
- Auto-refresh every 10 seconds

/internal/onboarding

- A guided workflow for onboarding a new client organization:
  Step 1: Create organization (name, type, account_type, access_code)
  Step 2: Create owner user account for the org (username, email, temp password)
  Step 3: Send welcome email to the new owner (uses existing email_provider.py templates)
  Step 4: Confirm setup (shows org ID, owner credentials summary, next steps checklist)
- Each step calls the existing API endpoints — this is a UI workflow, not new backend logic
- Checklist items for the new org admin: set branding, add team members, upload first
  catalog batch, connect Dropbox or Google Drive, configure notification preferences

Portal Frontend Implementation:

- The portal frontend is a single React component tree served under /internal
- Built with the same React 18 + Vite + Tailwind stack as the existing frontend
- Add an /internal route to the existing App.jsx router
- Use a separate layout component (InternalLayout.jsx) with its own sidebar and navigation
  distinct from the client-facing Sidebar.jsx
- The portal must be fully responsive — staff may be working from laptops or tablets
- Color palette: use a visually distinct palette from the client-facing UI so it is
  immediately obvious you are in an internal tool (suggested: dark navy sidebar, white
  content area — avoid the existing sage green/gold client palette)

### COMPONENT 6: OBSERVABILITY AND ERROR HANDLING

A production backend must surface errors, log meaningfully, and behave predictably under
failure conditions.

Structured Logging Enhancements:

- Extend the existing JSONFormatter in backend/utils/logging_config.py to include:
  - request_id: a UUID generated per request and propagated via middleware
  - user_id: extracted from JWT if present
  - org_id: extracted from request context if available
  - route: the matched FastAPI route path
  - duration_ms: request processing time in milliseconds
- Add a RequestLoggingMiddleware to backend/main.py that:
  - Generates a request_id UUID on every incoming request
  - Logs request start (method, path, user_id if authenticated)
  - Logs request end (status_code, duration_ms)
  - Stores request_id in a Python ContextVar so it is available to all downstream log calls

Error Handling:

- Add a global exception handler in backend/main.py using @app.exception_handler(Exception)
- In production (APP_ENV=production): return { “error”: “An unexpected error occurred”,
  “request_id”: “<request_id>” } with status 500 — never expose stack traces
- In development (APP_ENV=development): return full traceback in the response for
  debugging convenience
- Always log the full exception with traceback at ERROR level regardless of environment
- Add specific handlers for common cases:
  - SQLAlchemyError: log and return 503 with “Database error”
  - JWT decode errors: return 401 with “Invalid or expired token”
  - Pydantic ValidationError: return 422 with the validation details (this is FastAPI default
    behavior — ensure it is not overridden)

Request ID Propagation:

- Add X-Request-ID to every response header (value: the request_id UUID)
- This allows staff and engineers to correlate a specific failed request in the logs

### COMPONENT 7: DATABASE MIGRATION SAFETY

The existing Alembic setup runs migrations on every boot via db_setup.py. As the team
grows, migrations need to be safe and auditable.

Requirements:

- Ensure db_setup.py calls alembic upgrade head (not a raw CREATE TABLE) — if it
  currently does both, remove the raw DDL and let Alembic own schema management exclusively
- Add a migration lock: before running alembic upgrade head, check for a migration_lock
  table with a single row. If a lock row exists with status=“running” and
  started_at < NOW() - INTERVAL ‘10 minutes’, assume a crashed migration and clear it.
  Insert a lock row before migrating, remove it after. This prevents two simultaneous
  deployments from running migrations concurrently.
- Add a GET /api/internal/migration-status endpoint (requires is_master_admin) that returns:
  - Current Alembic revision
  - Pending migrations (if any)
  - Last migration timestamp
  - Last migration actor (hostname + timestamp)

## IMPLEMENTATION NOTES

### Do Not Break Existing Functionality

Every change must be backward compatible with the existing 49 route modules, 80 database
tables, and all frontend pages. Run the application and verify it boots cleanly after each
component is implemented. Do not modify existing route logic, model definitions (except
the is_cadence_staff addition via migration), or service implementations unless explicitly
required by one of the components above.

### Migration Strategy

- The is_cadence_staff field (Component 3) requires a new Alembic migration
- The UserSession table (Component 3) requires a new Alembic migration
- The migration_lock table (Component 7) should be created directly in db_setup.py as
  a bootstrap table (not via Alembic, since it needs to exist before Alembic runs)
- All other components are additive — new routes, new files, new frontend pages — and
  require no schema changes

### File Organization

New files created by this build should follow the existing project structure:

- New route modules → backend/routes/
- New service files → backend/services/
- New utility functions → backend/utils/
- New Alembic migrations → alembic/versions/
- New scripts → scripts/
- New frontend pages → frontend/src/pages/
- New frontend components → frontend/src/components/
- Documentation → project root (DEPLOYMENT.md) or docs/ subdirectory

### Environment Variable Additions (add all to Replit Secrets)

- APP_ENV — “development” or “production”
- ALLOWED_ORIGINS — comma-separated list of allowed CORS origins
- STAGING_DATABASE_URL — PostgreSQL connection string for staging DB
- DOCS_USERNAME — HTTP Basic Auth username for /docs and /redoc in production
- DOCS_PASSWORD — HTTP Basic Auth password for /docs and /redoc in production
- BUILD_VERSION — git commit hash or build tag (optional, used by GET /health)

### Code Quality Standards

- All new Python code must pass flake8 with default settings
- All new route handlers must include proper FastAPI type annotations on parameters
  and return types
- All new database models must follow the existing SQLAlchemy declarative pattern in
  backend/models/models.py
- All new React components must use functional components with hooks — no class components
- Tailwind utility classes only — no inline styles, no separate CSS files
