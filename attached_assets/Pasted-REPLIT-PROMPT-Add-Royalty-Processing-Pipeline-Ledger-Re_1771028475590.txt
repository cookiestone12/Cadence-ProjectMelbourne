PART B — ROYALTY PROCESSING ENGINE (SERVER LOGIC)
════════════════════════════════════════════════════════════

Add an async worker/queue for:
- parsing statement into royalty_statement_lines
- auto-matching lines to songs/works/releases
- processing statement into ledger entries

Key flows:
1) Upload -> Parse -> Create statement + lines
2) Column mapping (if needed) -> update lines fields
3) Matching (auto + manual confirmation)
4) Process statement -> create ledger entries:
   For each CONFIRMED/AUTO_MATCHED line:
   a) Determine song/work/release
   b) Determine applicable contracts/splits for revenue_type
   c) Allocate earning per payee
      - Create EARNING entries per payee
   d) Apply fees (if you track mgmt/admin fees) as FEE entries
   e) Apply recoupment:
      - Find active advances for payee, ordered by recoupment_priority
      - Only apply if advance.recoupable=true and pool matches revenue_type (MASTER->MASTER, etc.)
      - applied = min(net_payable, outstanding_balance)
      - Create RECOUPMENT_APPLIED entry for applied amount
      - Decrement advance_balances.outstanding_balance
   f) Remaining = payable
      - Create PAYABLE_CREATED entry
5) Lock statement after processing:
   - status -> PROCESSED then LOCKED
   - store processing_run_id and version

Reprocessing:
- Create new processing run with incremented run_version
- Create REVERSAL ledger entries to negate all prior run’s non-reversal entries (preserve audit)
- Run processing again and generate new entries

Do not allow deleting processed statements or ledger entries.
