Add a comprehensive date-stamped edit history to song records so every change 
to splits, contributors, and metadata creates a digital trail. The goal is to 
turn each song into a tracked asset with a verifiable history of who 
contributed what and when.

DATA MODEL:
1. Create a new table `song_edit_history` with these columns:
   - id (UUID PK)
   - song_id (FK to songs.id)
   - organization_id (FK)
   - user_id (FK to users.id — who made the edit)
   - field_name (string — e.g. "splits", "credits", "release_date", "title")
   - old_value (JSONB)
   - new_value (JSONB)
   - change_type (enum: 'create' | 'update' | 'delete' | 'add_contributor' | 
     'remove_contributor' | 'split_change')
   - notes (text, nullable — optional user-supplied reason)
   - created_at (timestamp)

   Add migration in db_setup.py.

BACKEND:
2. In songs.py, credits.py, and any route that mutates a song or its 
   credits/splits, write a song_edit_history record on every mutation. 
   Capture the before and after JSON of the affected fields. Use a helper 
   function in utils/ (create utils/edit_history.py) so this is consistent 
   across routes.

3. Add a new endpoint GET /api/songs/{song_id}/history that returns the 
   full edit history for a song, ordered by created_at descending, with 
   user info joined.

4. The audit_log table already exists for org-wide actions — keep that 
   separate. song_edit_history is specifically for per-song asset 
   provenance and should be more granular.

FRONTEND:
5. On SongDetail.jsx, add a new "History" tab alongside the existing tabs 
   (Metadata, Credits, Splits, Audio, Analytics, etc.). The History tab 
   shows a chronological timeline of every edit: who made it, when, what 
   field changed, and the before/after values. Use a clean timeline UI 
   pattern (vertical line with dated entries).

6. Each timeline entry should be expandable to show the full diff for 
   that change.

7. When a user edits splits or adds/removes a contributor, prompt them with 
   an optional "Reason for change" text field that gets saved to the 
   edit_history.notes column. This is optional — they can skip it — but 
   encouraged for splits changes.

8. On the song detail page header, show a small "Last edited [date] by 
   [user]" indicator next to the song title.

EXPORT:
9. Add a "Download Edit History" button on the History tab that exports the 
   timeline as a PDF for that single song. Useful for disputes or audits. 
   Use the existing PDF generation pattern from registration_reports.py 
   as a reference.

DO NOT touch catalog UI navigation, splits normalization, or royalty 
processing in this prompt. The edit history table should populate going 
forward from deployment — no backfill of historical data needed. Confirm 
existing song CRUD tests still pass.
