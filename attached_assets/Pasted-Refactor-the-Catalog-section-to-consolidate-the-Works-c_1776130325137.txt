Refactor the Catalog section to consolidate the Works concept under Catalog 
and simplify the released/unreleased mental model.

NAVIGATION CHANGES:
1. In the left sidebar, rename the "Works" tab to "Unreleased" and nest it 
   as a subtab under "Catalog". Update App.jsx routing so /works still 
   resolves but the canonical route is /catalog/unreleased. Keep WorksPage.jsx 
   as the underlying component for now — just rename the user-facing label and 
   reposition it in navigation.

2. In NewCatalogPage.jsx, change the top-level filter tabs from 
   [All | Recordings | Compositions] to [All | Released | Unreleased]. 
   Drop "Compositions" as a top-level distinction entirely. A song's bucket 
   is determined by its release status, not by whether it's a composition vs. 
   recording.

3. Within the Unreleased view, add a secondary label/filter for entry type: 
   Song, Instrumental, Remix, Sample, Demo. This is metadata on the song 
   record, not a separate tab. Add a `entry_type` enum column to the songs 
   table via a migration in db_setup.py.

DATE-TRIGGERED RELEASE STATUS:
4. Add a `release_date` (nullable date) and `release_status` (enum: 
   'unreleased' | 'released', default 'unreleased') to the songs table. 
   Migration in db_setup.py.

5. When a user creates or edits a song and sets a release_date, store it. 
   On the backend, add a lightweight scheduled job to email_scheduler.py 
   (APScheduler) that runs daily at 00:05 UTC and flips any song where 
   release_date <= today AND release_status = 'unreleased' to 'released'. 
   Log each flip to the audit_log table.

6. Allow manual override: a "Mark as Released" button on SongDetail.jsx 
   that immediately flips status regardless of date. Same audit log entry.

7. New songs default to release_status = 'unreleased' and land in the 
   Unreleased view automatically. Remove any "unfiled" intermediate state — 
   the Unreleased bucket IS the holding area.

DUPLICATE SONG FEATURE:
8. On SongDetail.jsx, add a "Duplicate" action that creates a copy of the 
   song with " (Copy)" appended to the title, same credits and splits, 
   release_status reset to 'unreleased', and a new field `parent_song_id` 
   (nullable FK to songs.id) set to the original song's ID. This supports 
   remix/instrumental/EP-variant workflows without re-entering metadata.

9. On the duplicated song's detail page, show a "Linked to: [original 
   song title]" reference that links back to the parent.

DO NOT touch the royalty engine, splits normalization, or audit trail logic 
in this prompt — those are separate updates. Do not delete WorksPage.jsx. 
Do not remove existing /works route, just deprecate it in favor of 
/catalog/unreleased. Confirm all existing tests pass before finishing.
